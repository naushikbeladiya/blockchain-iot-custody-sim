1. Executive Summary
This PRD specifies the requirements for a discrete-event simulation (DES) framework that models a three-tier blockchain–IoT cryptographic chain-of-custody system for last-mile parcel delivery. The simulation must reproduce all quantitative results presented in the research paper “Cryptographic Chain-of-Custody for Last-Mile Parcel Delivery” (Beladiya, 2026), specifically the three core data tables:
	•	Table 2: Tamper Detection Rate (TDR) by attack category — 99.96% aggregate across 50,000 handoffs
	•	Table 3: End-to-end confirmation latency by blockchain configuration — 3 configs × 3 percentiles
	•	Table 4: Gas consumption comparison: naïve vs. edge-optimized — 25.1% total reduction

Primary deliverable: A Python package (installable via pip) with a CLI that runs the complete simulation and outputs CSV files, console summaries, and optional matplotlib charts matching the paper’s results.
Secondary deliverable: A public GitHub repository with README, reproducibility instructions, MIT license, and GitHub Actions CI that runs the simulation on every push.
Why this matters: The research paper has been submitted to IET Blockchain (BLC-2026-04-0069). Providing a working, reproducible simulation significantly increases the probability of acceptance by demonstrating empirical rigor and enabling reviewers to verify claims independently.

2. Project Context & Objectives
2.1 Background
The research paper proposes a system where IoT cryptographic seals attached to parcels generate signed custody events at each handoff (warehouse → carrier → consumer). Events are verified on-chain by smart contracts that check ECDSA signatures, recompute SHA-256 hashes, validate timestamps, enforce business rules, and record custody transitions immutably. The simulation models this pipeline without requiring a live blockchain deployment.
2.2 Objectives
	•	Reproduce all three quantitative tables from the paper with statistically consistent results (within ±5% of reported values across multiple random seeds)
	•	Provide a configurable, parameterized simulation that researchers can extend and modify
	•	Generate publication-quality output (CSV data, LaTeX-formatted tables, matplotlib figures)
	•	Pass automated validation tests confirming that results match paper claims
	•	Be fully self-contained: no external blockchain infrastructure, no API keys, no cloud services required
2.3 Non-Goals
	•	Live deployment on Ethereum mainnet, testnets, or any real blockchain
	•	Actual IoT hardware integration or firmware
	•	Web-based dashboard or GUI (CLI + CSV output is sufficient)
	•	Real-time monitoring or streaming visualization
	•	Smart contract Solidity code (contracts are modeled in Python, not deployed)

5. Module Specifications
5.1 Network Topology Module
File: src/custody_sim/topology.py
Purpose: Defines the simulated logistics network and blockchain validator set.

Classes:
	•	DistributionCenter(id, location) — 5 instances
	•	TransitHub(id, location) — 12 instances
	•	Courier(id, location, capacity) — 48 instances
	•	Validator(id, stake) — configurable: 4, 8, 16, or 32 instances
	•	NetworkTopology — factory class that builds the complete network from a YAML config

Parcel routing: Each parcel is assigned a route consisting of 2–5 nodes (drawn from a truncated Poisson distribution with mean 3.2). The route always starts at a DistributionCenter and ends at a simulated consumer endpoint, with intermediate TransitHubs and Courier nodes.
Communication latency: Each network link has a stochastic latency drawn from a log-normal distribution (mean 200ms, std 100ms) to model cellular transmission delays from IoT devices to the blockchain mempool.
5.2 IoT Seal Device Simulator
File: src/custody_sim/iot_seal.py
Purpose: Simulates the Tier 1 IoT cryptographic seal device behavior at each custody handoff.

Class: IoTSeal(device_id, private_key, public_key)
For each handoff event, the IoT seal executes:
	•	generate_telemetry() — returns a dict with temperature (uniform 2–8°C for pharma, 15–25°C for general), humidity (30–70% RH), shock (0–2g normal, >5g = anomaly), GPS coordinates (lat/lon perturbed from route waypoints)
	•	encrypt_payload(telemetry) — AES-256-GCM encryption of JSON-serialized telemetry with a per-session symmetric key. Returns ciphertext bytes.
	•	construct_message(encrypted_payload, timestamp, sender_id, receiver_id, seq_num) — canonical serialization using struct.pack with network byte order (big-endian), length-prefixed fields
	•	compute_hash(message) — SHA-256 of the canonical byte sequence. Returns 32-byte digest.
	•	sign_hash(digest) — ECDSA signature over secp256k1 using the device’s private key. Returns (r, s, v) signature components.

Output: A CustodyEvent dataclass containing: encrypted_payload, timestamp, sender_id, receiver_id, device_id, sequence_number, composite_hash, signature
5.3 Cryptographic Protocol Engine
File: src/custody_sim/crypto_engine.py
Purpose: Provides all cryptographic primitives used across the simulation.

Functions:
	•	generate_keypair() — generates ECDSA secp256k1 key pair, returns (private_key, public_key)
	•	sha256(data: bytes) -> bytes — SHA-256 hash
	•	ecdsa_sign(private_key, digest) -> Signature — ECDSA signing
	•	ecdsa_verify(public_key, digest, signature) -> bool — ECDSA verification (equivalent to ecrecover)
	•	aes_encrypt(key, plaintext) -> bytes — AES-256-GCM encryption
	•	aes_decrypt(key, ciphertext) -> bytes — AES-256-GCM decryption
	•	canonical_serialize(fields: list) -> bytes — deterministic length-prefixed serialization

Critical requirement: All crypto operations must use REAL cryptographic primitives from the cryptography library — not mocks or simulated hashes. The simulation’s security claims depend on actually executing SHA-256 and ECDSA.
5.4 Smart Contract Verifier (Python Model)
File: src/custody_sim/contract_verifier.py
Purpose: Models the Tier 2 smart contract verification logic in Python. This is NOT Solidity — it’s a Python class that implements identical logic.

Class: DeviceRegistry
	•	register_device(device_id, public_key) — adds device to authorized registry
	•	get_public_key(device_id) -> public_key — lookup
	•	is_active(device_id) -> bool — checks operational status

Class: CustodyVerifier
verify(event: CustodyEvent)The 
	•	Device authentication: Look up device_id in DeviceRegistry; reject if not found or inactive
	•	Signature verification: Call ecdsa_verify(public_key, event.composite_hash, event.signature); reject if False
	•	Hash integrity: Recompute hash from (encrypted_payload || timestamp || context); reject if != event.composite_hash
	•	Timestamp validation: Reject if |event.timestamp - current_block_timestamp| > SKEW_WINDOW (default 300s)
	•	Sequence validation: Reject if event.sequence_number <= last_recorded_sequence for this (device_id, asset_id) pair
	•	Business rules: Evaluate configurable rules (temperature in [2,8]°C, GPS within geofence radius, actor sequence authorized)
	•	State recording: Append verified event to AssetLedger; emit CustodyTransferred or AnomalyDetected event

Return value: VerificationResult(success: bool, failure_reason: Optional[str], gas_used: int)
Gas estimation: Each verification step has a fixed gas cost (see Section 5.6). The total gas is the sum of costs for all steps executed before either success or first failure.
5.5 Adversarial Attack Engine
File: src/custody_sim/adversary.py
Purpose: Generates malicious custody events across 5 attack categories for security evaluation.

Class: AdversarialEngine(injection_rate=0.10)
The injection_rate parameter controls what fraction of total events are adversarial (default 10%). For each adversarial event, the engine randomly selects one of five attack types with the following distribution:

Attack implementations:
	•	modify_payload: Randomly flip 1–4 bytes in the encrypted_payload while keeping the original signature and hash unchanged. The hash mismatch will be detected at Step 3 of verification.
	•	manipulate_timestamp: Set the timestamp to a value outside the skew window (e.g., current_time ± 600s) while preserving the original signature. The timestamp check at Step 4 will detect this.
	•	forge_signature: Generate a new ECDSA signature using an unauthorized private key (not registered in DeviceRegistry). Step 2 will detect this.
	•	replay_event: Re-submit a previously legitimate, successfully verified event with the original timestamp and signature intact. Steps 4 and 5 (timestamp window + sequence number) should catch this, except in the edge case described in the paper.
	•	violate_sequence: Submit an event with a sequence number equal to or less than the last recorded value for the (device_id, asset_id) pair. Step 5 will detect this.

Replay attack edge case: The simulation must model the specific scenario where 2 out of 1,000 replay attacks succeed: this occurs when network latency causes the replayed event to arrive within the skew window AND before the original event’s block has propagated. This requires precise coordination between the GasCostModel’s block propagation delay and the adversary’s replay timing. The simulation should produce 998/1000 detection at ±300s skew and 1000/1000 at ±120s skew.
5.6 EIP-1559 Gas Cost Model
File: src/custody_sim/gas_model.py
Purpose: Models Ethereum’s EIP-1559 fee mechanism for transaction cost estimation and block-level fee dynamics.

Class: EIP1559GasModel(base_fee_initial, elasticity_multiplier=2, target_utilization=0.5)

Base fee adjustment algorithm (per block):
if block_utilization > target_utilization:
    base_fee_next = base_fee * (1 + 0.125 * (utilization - target) / target)
else:
    base_fee_next = base_fee * (1 - 0.125 * (target - utilization) / target)

# Capped at 12.5% change per block in either direction

5.7 Metrics Collection & Reporting
File: src/custody_sim/metrics.py
Purpose: Collects all simulation data and produces the output tables, CSVs, and optional charts.

Class: MetricsCollector
Continuously records every event with:
	•	event_id, event_type (legitimate/adversarial), attack_category (if adversarial)
	•	device_id, asset_id, sender_id, receiver_id
	•	timestamp_created (device-level), timestamp_submitted (to mempool), timestamp_confirmed (in block)
	•	verification_result (pass/fail), failure_reason
	•	gas_used_naive, gas_used_optimized
	•	block_number, block_config (ethereum/l2/high-throughput)

Output methods:
	•	generate_tdr_table() — produces Table 2 (TDR by attack category)
	•	generate_latency_table() — produces Table 3 (latency percentiles by config)
	•	generate_gas_table() — produces Table 4 (gas comparison)
	•	export_csv(directory) — writes all raw data + summary tables to CSV
	•	export_latex(directory) — writes LaTeX-formatted tables
	•	plot_charts(directory) — generates matplotlib figures (latency CDF, TDR bar chart, gas breakdown)

6. Data Models & Schemas
6.1 CustodyEvent
@dataclass
class CustodyEvent:
    event_id: str              # UUID
    asset_id: str              # Parcel identifier
    device_id: str             # IoT seal device ID
    sender_id: str             # Transferring party
    receiver_id: str           # Receiving party
    timestamp: float           # Unix timestamp (ms precision)
    sequence_number: int       # Monotonically increasing per (device, asset)
    encrypted_payload: bytes   # AES-256-GCM encrypted telemetry
    composite_hash: bytes      # SHA-256(encrypted_payload || timestamp || context)
    signature: bytes           # ECDSA signature over composite_hash
    is_adversarial: bool       # Ground truth label
    attack_type: Optional[str] # None if legitimate
6.2 VerificationResult
@dataclass
class VerificationResult:
    success: bool
    failure_reason: Optional[str]  # None if success; else one of:
    # 'device_not_registered', 'device_inactive', 'signature_invalid',
    # 'hash_mismatch', 'timestamp_expired', 'sequence_violation',
    # 'business_rule_temperature', 'business_rule_geofence',
    # 'business_rule_actor_sequence'
    gas_used_naive: int
    gas_used_optimized: int
    verification_time_ms: float
